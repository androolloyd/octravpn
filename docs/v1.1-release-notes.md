## What changed since v1

### Cryptographic equivocation slashing (`slash_double_sign`)

The 2026-05-14 Octra dev-team announcement confirmed `ed25519_ok(pk,
msg, sig) -> bool` (mainnet reference contract
`octBDvZSiTqdEBAyFSp79CHeoLMR9MzHugX9YkHtuQ57MRB`). v1.1 uses it to
add a second slash path on top of the in-AML equivocation already in
`settle_claim`.

| Path | Detector | Evidence | Caller |
|---|---|---|---|
| `settle_claim` equivocation (v1) | The AML itself — it sees both `settle_claim` txs and compares `bytes_used` | On-chain only | Operator's own second submission |
| `slash_double_sign` (v1.1, new) | Off-chain dual-sig protocol catches operators who sign two contradictory receipts to two different clients but never re-claim on chain | Two ed25519-signed receipt payloads + sigs | Any party with the receipts |

The first protects against operators who try to game on-chain
settlement. The second protects against operators who try to game
off-chain receipt protocols (e.g., billing two clients differently
for the same session-seq).

Slash payout is identical: 90 % burn to program treasury, 10 %
bounty to the caller.

### `EndpointRecord` carries `receipt_pubkey`

`register_endpoint` now takes a 7th positional parameter — the
operator's ed25519 receipt-signing pubkey (hex). `octravpn-node`
auto-fills it from the HKDF-derived receipt key (same key already
used to sign off-chain receipts in `octravpn-core::receipt`).
**No operator-side config change** required — running
`octravpn-node register` against a v1.1 program just works.

### Client-side `slash-evidence submit`

`octravpn slash-evidence` already had `verify` and `build`. v1.1
adds:

```sh
octravpn slash-evidence submit <evidence.json>
```

Loads the JSON evidence blob, re-verifies signatures locally, then
submits `slash_double_sign` on chain. If the slash succeeds the
caller's wallet receives the 10 % bounty.

End-to-end flow for a slasher who's collected two contradictory
receipts:

```sh
# 1. Verify the blob is well-formed.
octravpn slash-evidence verify ./equivocation.json

# 2. Submit on chain. Bounty lands in your wallet on success.
octravpn slash-evidence submit ./equivocation.json
```

## Backward compatibility

- Existing v1 deployments stay valid until the v1.1 program is
  deployed at a fresh address. There's no migration path inside one
  program instance — the on-chain `EndpointRecord` schema changed.
- Operators registered against a v1 program continue to work on
  that program. To benefit from `slash_double_sign` they must
  re-register against a v1.1 deployment with the receipt_pubkey.
- `octravpn-node` is forward-compatible: it always passes the
  receipt_pubkey, so it works against both v1 and v1.1 programs
  (v1's `register_endpoint` parser may or may not accept the extra
  trailing param depending on how strictly the AML enforces param
  count — verify against your target deployment).

## What's NOT in v1.1 (still v2)

- Hidden-operator identities (Circles)
- Per-class ACL routing (shared exit vs internal subnet)
- Encrypted byte counters at settlement
- Multi-hop session escrow

These remain blocked on the Octra Circle DSL + proxy/access-contract
grammar, both of which the dev team has indicated are coming soon.
Design captured in `docs/v2-circles-design.md`.

# OctraVPN — External Security Audit Orientation

> Snapshot: see `manifest.json` for SHA-256 hashes and the commit ID
> this package was generated from. All file paths in this directory
> are repo-relative.

This directory is the audit-prep package for an external review of
the OctraVPN codebase. It is intentionally read-only: every artifact
under `docs/audit/` is generated by a deterministic command described
inline or in the sibling `manifest.json`, and can be regenerated by
an auditor who clones the repo at the same commit.

---

## 1. What OctraVPN is

OctraVPN is a Rust-based mixnet-style VPN whose **session
accounting** lives on the public Octra blockchain. Sessions are
opened, byte-counted, and settled via on-chain `contract_call` ops
into an AML smart-contract program (`program/main-v3.aml`). The wire
layer is WireGuard (boringtun); coordination uses a forked
Tailscale-style control plane (`headscale-api`, sibling repo). The
crypto-economic claim is:

- **Operators are bonded.** Equivocation on a dual-signed receipt is
  publicly slashable (`slash_double_sign` in the AML program).
- **Members are anchored.** Tailnet membership is the SHA-256 of a
  canonical off-chain `members.json`, anchored on chain so the owner
  cannot quietly rewrite history.
- **Receipts are dual-signed.** A client + operator both sign every
  `(session_id, seq, bytes_used)` tuple, with a v1.2 domain binder
  over `(program_addr, chain_id, circle_id)` to prevent cross-deploy
  replay.

See `docs/architecture.md` and `docs/v3-circle-resident-architecture.md`
for the design narrative, and `docs/audit/threat-model-summary.md`
in this directory for the 2-page exec summary.

## 3. Build instructions for the auditor

```sh
# 1. Toolchain pinned by rust-toolchain.toml
rustup show               # picks up the channel automatically

# 2. Compile everything (workspace)
cargo build --workspace --all-targets

# 3. Run the unit + property test suites
cargo test  --workspace --all-targets

# 4. Lean proofs (optional — needs Lean 4 ≥ 4.x from lean-toolchain)
cd proofs/lean && lake build

# 5. Reproduce the devnet stack (Docker required; never run nodes natively)
cd docker/devnet
cp hosts.env.example hosts.env   # edit as needed
docker compose -f docker-compose.devnet.yml up -d --build
./preflight.sh
./e2e.sh                         # happy-path
./e2e-adversarial-v3.sh          # v3 adversarial drill
```

`scripts/security/redteam.sh` is a one-shot driver that runs every
`e2e-adversarial-*.sh` drill, the full fuzz suite for a bounded
duration, and a curl-based RPC fuzzer; it exits non-zero if any
defense fails. Treat it as the canonical "did the audit-prep package
hold up under attack" CI gate.

## 4. Scope

### In scope (assume these are attacker-reachable)

- **Wire protocol** — WG handshake handling, onion peel, dual-signed
  receipts (`crates/octravpn-core/src/{tunnel.rs,onion.rs,receipt.rs}`).
- **Control plane** — HTTP `/session` POST + GET endpoints, rate
  limiter, audit log, receipt journal
  (`crates/octravpn-node/src/{control.rs,audit.rs,rate_limit.rs}`,
  `crates/octravpn-core/src/receipt_journal.rs`).
- **Chain interaction** — JSON-RPC client (rustls), contract_call
  envelope construction, settlement glue
  (`crates/octravpn-core/src/{rpc.rs,v3_calls.rs}`,
  `crates/octravpn-node/src/chain_v3.rs`,
  `crates/octravpn-client/src/{chain_v3.rs,settler.rs}`).
- **Sealed assets / key handling** — `octravpn-node seal-keys`, sealed
  policy + members + state-root anchors, key encryption envelope
  (`crates/octravpn-node/src/seal.rs`,
  `crates/octravpn-core/src/{v3_state_root.rs,v3_policy.rs,v3_members.rs,v3_canonical.rs}`).
- **ACL evaluation** — canonical ACL parser, on-chain hash anchor,
  default-deny match engine (`crates/octravpn-mesh/src/acl.rs`).
- **Preauth + Tailscale wire bridge** — preauth minter, headscale-api
  bridge (`crates/octravpn-mesh/src/headscale_bridge.rs`,
  `crates/octravpn-node/tests/tailscale_wire_integration.rs`). The
  Tailscale wire-protocol bytes themselves live in the sibling
  `../headscale-rs/headscale-api/src/tailscale_wire/` crate and are
  in scope for this audit (re-exported as
  `octravpn_mesh::tailscale_wire`).
- **AML program** — `program/main-v3.aml`'s slash, bond, settle,
  pause, and governance entrypoints, and the v3 anchor schemas they
  commit to.
- **Lean spec / Rust binding** — invariants claimed in
  `proofs/lean/{WireProtocol,OctraVPN_Rust,OctraVPN_V2}/*.lean` MUST
  match the Rust they model.

### Out of scope

- **Operator deployment / k8s manifests / systemd units** —
  `deploy/`, `docs/operators/`. These ship hardening guidance; an
  operator that ignores it accepts the residual risk.
- **Third-party hardware wallets** — we never see private keys; key
  custody is the operator's responsibility.
- **The Octra chain itself** — devnet `devnet.octrascan.io` is
  treated as an untrusted RPC endpoint (no cert pinning by default,
  see security-properties §13). Its consensus, AML interpreter, and
  storage layer are NOT in this audit's scope, but assumptions we
  make about them ARE (e.g. the chain delivers `slash_double_sign`
  atomically — if it doesn't, equivocation is undetectable, and
  that is documented as a precondition).
- **Native (non-Docker) node runs** — the test harness is
  Docker-only by user policy (see `feedback_docker_only.md` memory).
  Behaviour observed by running `cargo run -p octravpn-node` natively
  is not a supported attack surface — but the binary itself should
  still be safe in isolation, and the fuzz targets exercise that.
